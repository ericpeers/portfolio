# Parallelism in Portfolio

## Call Chain

When a request arrives at `/users/:id/glance` or `/portfolios/compare`, the hot path runs through `ComputeDailyValues` in `performance_service.go`. For each security in the portfolio, it must fetch a time series of daily prices — either from the PostgreSQL cache or, on a cache miss, from the EODHD API. `ComputeDailyValues` spawns one goroutine per security immediately, then uses a buffered-channel semaphore (`priceConcurrency`, default 20) to limit how many goroutines proceed concurrently into `GetDailyPrices`. Inside `GetDailyPrices`, a cache hit returns quickly after two DB queries (price range check, then price rows). A cache miss falls through to `fetchAndStore`, which acquires a second semaphore (`fetchSem`, default 10) before opening an EODHD connection. Within that connection, prices are fetched first, then splits and dividends are fetched in two parallel goroutines via `GetStockEvents`. Every outbound EODHD call — whether for prices, splits, or dividends — passes through a shared token-bucket rate limiter before the HTTP request fires. Results are collected over a channel; the first error cancels the derived context so remaining goroutines exit without making further network calls.

## Selected Values

The EODHD rate limiter is configured at 16 requests per second (960/min), keeping throughput just under EODHD's 1000/min limit with even distribution across the minute. The token bucket uses continuous proportional replenishment rather than a lump per-interval refill, so the bucket never accumulates more than 16 tokens and no burst is possible — this avoids the sliding-window 429s that a per-minute burst strategy triggers. `fetchSem` is set to 10: by Little's Law, saturating 16 req/sec at roughly 300ms average EODHD latency requires only ~5 concurrent connections, so 10 is comfortably right-sized without over-committing sockets. `priceConcurrency` is intentionally set higher at 20: on warm-cache runs (the common case after first load) no provider connection is needed and 20 parallel DB reads measurably outperform 10, while on cold-cache runs the inner `fetchSem` becomes the effective ceiling and the extra goroutines simply wait on the channel at negligible cost.
